1. System Architecture Overview

The system is an asynchronous high-performance calculation engine for algorithmic stock screening and backtesting.

Layer Technology Stack & Specifications
Backend HTTP API FastAPI (Python 3.10+). Runs asynchronously on Uvicorn. Routes mounted strictly for JSON APIs and Static asset delivery. Multi-process load balancing via Gunicorn allowed.
Data Calculation Engine pandas_ta and numpy. Extensive DataFrame matrix calculations relying on vectorized shifting.
Databases (MySQL) Primary App Connection Pool: Local App State. Handles logic (signals, profiles, user data).
Secondary Datamart Pool: View schemas for read-only Universe sets, financial parameters, and indices.
Asynchronous DB Connector aiomysql pool managing concurrent execution loops per symbol map. Strict locking on scenario engine bounds.
Frontend Framework Strictly decoupled constraints: 0% JS Frameworks (No React/Vue). Pure Vanilla JS State implementation. 100% custom SVG rendering engine built into `app.js`. CSS Grid + Flexbox based theming.

3. Indicator Engine: Mathematical Logic

The engine resides in indicator_engine.py. It manages process_profile(pool, datamart_pool, profile_id, timeframe) mathematically mapping Pandas vectors.

Exact Indicator Constraints

  • Intraday Setup: RSI(14) with OB/OS bounds at 80/20. EMA (Fast=9, Slow=21). Supertrend (Len=10, Mult=2.5). Volume Period=20, Spike Ratio=1.5x.
  • Swing Setup: RSI(14) with OB/OS bounds at 70/30. EMA (Fast=9, Slow=20). Supertrend (Len=10, Mult=3.0). Volume Period=20, Spike Ratio=2.0x.
  • DMA Anchoring: Simple Moving Averages (10, 20, 50, 200) are explicitly anchored to the 1D timeframe regardless of what intraday subset is executed. The engine uses memory caching shared_cache to stop identical pandas recalculation loops against 1D base data.

Resampling Data Injection ("Live Candle Synthesis")

Critical Architectural Detail: The 1 Day (1d) timeframe data is historically static till End-Of-Day. To bypass this limitation, synthesize_live_candle() takes the MAX(timestamp) from the app_sg_ohlcv_prices WHERE timeframe='5m' table. If the day matches the current day, it explicitly constructs a 1-day candle (Open = First 5m Open, High = Max(5m Highs), Low = Min(5m Lows), Close = Last 5m Close, Volume = Sum(5m Volumes)) and prepends this to the Historical dataset.

Confluence Weighting Matrix Algorithm

An aggregate integer score dynamically determining multi-factor signal strength. Values sum up to rank between -5 and +5.

# Strict Integer Algorithm
rank = 0
if ema_signal == 'BUY': rank += 1 elif ema_signal == 'SELL': rank -= 1
if st_dir == 'BUY': rank += 1 elif st_dir == 'SELL': rank -= 1
if rsi_val > 50: rank += 1 elif rsi_val < 50: rank -= 1
if ltp > dma_data.get('SMA_20'): rank += 1  else: rank -= 1
if vol_signal == 'BULL_SPIKE': rank += 1 elif vol_signal == 'BEAR_SPIKE': rank -= 1

# Strict "PERFECT" Qualification Rule
if rank >= 4 and vol_signal == 'BULL_SPIKE' and ema_signal == 'BUY' and st_dir == 'BUY':
    trade_strategy = "PERFECT_BUY"

4. Scenario Engine (Backtesting)

Managed in scenario_engine.py. This handles temporal merging of varying time constraints (e.g. 5m execution triggered by 15m structural breaks).

Look-Ahead Bias Protection Mechanism

When aggregating Multi-Timeframe Data points, the mathematical merging algorithm applies a strict lag vector index to guarantee the high-scale bar actually "closed" prior to evaluating its value against the low scale base execution.

# Merging Math applied over DataFrame 'df_primary_subset' derived initially from TF=15m and applied over Base tf=5m:
df_primary_subset[['RSI', 'ST_primary_dir', 'ST_primary_val']] = df_primary_subset[['RSI', 'ST_primary_dir', 'ST_primary_val']].shift(1)

df_combined = pd.merge_asof(
    df_base, 
    df_primary_subset, 
    on='timestamp', 
    direction='backward'
)

Tranche Loading Algorithms

Backtest algorithm permits user scaling. Exact array mapping used internally in simulation bounds:

  • Entry Scaling: Controlled by array `tr_weights = [W1, W2, W3]` matching Array Drop `tr_prices=[P1, P2]`. E.g: Weight 50%,25%,25% dropping 1%,2%.
  • Averaging Math: Avg_Entry = ((entry1 * w1) + (entry2 * w2) + (entry3 * w3)) / (w1+w2+w3)
  • Exit Targets: `t1_weight` vs `t2_weight`. The engine will sequentially strip off percentage weights upon target triggers or structurally on upper-level SuperTrend breaches. P&L calculates the % against the current weight subset only.

Server-Sent Events (SSE) Streaming

During live long-polling calculation runs, standard HTTP calls timeout. The infrastructure explicitly mounts /api/stream/fetch-data utilizing FastAPI's StreamingResponse(event_generator, media_type="text/event-stream") construct.

The Python event stream yields chunks in format: yield f"data: {json_string}\n\n" while generating. The frontend intercepts these chunks instantly utilizing fetchEvtSource = new EventSource(...) to immediately manipulate the DOM's progress bar (document.getElementById('sys-prog-bar-(swing/intraday)')) sequentially from 0% to 100%.
